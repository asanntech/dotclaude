# Commit Skill

## 処理フロー

1. **変更の確認** — `git status` と `git diff` で未コミットの変更を確認
2. **変更がない場合** — 「コミットすべき変更がありません」と伝えて終了
3. **ステージング状態の判定**:
   - **staged な変更がある場合** → staged のファイルのみをコミット対象とする（unstaged/untracked は無視）
   - **staged な変更がない場合** → 全変更をステージングしてコミット対象とする
4. **コミット対象の分析** — 対象ファイルの変更内容を分析
5. **コミットメッセージの生成** — 変更内容に基づいてコミットメッセージ（タイトルのみ）を生成
6. **ユーザーに確認** — 生成したコミットメッセージをユーザーに提示し、承認を求める
7. **コミットの実行** — ユーザーが承認したらコミットを実行

## コミットメッセージのルール

- **タイトルのみ**（本文は不要）
- **英語で記述**
- **Conventional Commits 形式**: `feat:`, `fix:`, `docs:`, `refactor:`, `test:`, `chore:` などの prefix を使用
- **70文字以内**
- **変更の「なぜ」を簡潔に表現**

## コミット実行時の注意

- staged な変更がある場合は追加の `git add` をしない（ユーザーの意図を尊重）
- staged な変更がない場合のみ `git add` でステージングする
- 機密ファイル（`.env`, `credentials.json` など）はステージングしない
- `Co-Authored-By` ヘッダーは付与しない
- コミット後に `git status` で結果を確認

## ユーザーへの確認フォーマット

```
以下のコミットメッセージでコミットします:

  <コミットメッセージ>

対象ファイル:
  - file1.ts (modified)
  - file2.ts (new)
```
